### Changed
- **`documentation-writer` bumped to 0.5.0 — mandatory output-format negotiation + UML skill delegation.** Driven by recurring user friction: previously the agent silently defaulted to Markdown, ignoring requests for LaTeX/PDF/DOCX even when the toolchain was available. New behaviour: a non-skippable Step 0 — Output format negotiation runs before any code reading or drafting. The agent probes the local toolchain (`pandoc`, `pdflatex`, `lualatex`, `xelatex`, `wkhtmltopdf`), builds the available-formats list (`md` always available; `tex`/`html`/`docx` need pandoc; `pdf` needs pandoc + a LaTeX engine OR `wkhtmltopdf` as fallback), surfaces the menu to the user with install hints for missing tools, and waits for an explicit answer (single format, list, `all`, `default` = `md+tex+html+pdf`, or `just md`). Default deny on unavailable formats — never silently degrade. Authors once in Markdown and converts to all agreed formats via pandoc through a documented single-source pipeline. The final deliverable surfaces produced files, line/page counts, and rendered diagrams referenced from `docs/diagrams/`. Bash tool added to dependencies (for the toolchain probe). Adds two skill dependencies: `doc-expert` and `uml-diagram-generator`. Diagrams are rendered artefacts, never inline source — pasting raw PlantUML/Mermaid into a `.md`/`.tex` file is now explicitly forbidden in the writing rules.
- **`doc-expert` bumped to 1.1.0 — format-aware authoring + diagram delegation.** Documentation principles section gains two new bullets: (1) format-aware authoring (use Markdown extensions that round-trip through pandoc — fenced code, tables, math via `$...$`, footnotes, cross-refs; avoid HTML inlines or framework-specific shortcodes that break on conversion); (2) mandatory delegation of diagrams to `uml-diagram-generator` — pasting raw PlantUML or Mermaid into the documentation source is forbidden because it does not render in PDF/DOCX without diagram-aware renderers and breaks the multi-format pipeline.

### Changed
- **`refactoring-supervisor` bumped to 3.0.0 — capability renamed "Application Refactoring" → "Application Replatforming"; rewriting phase fully redesigned as a progressive, incremental, test-driven, continuously validated 7-step loop.** Major version bump because the workflow shape changes: there is no longer a separate Phase 5 (TO-BE Testing & Equivalence Verification) — final validation is now Step 6 of the new Phase 4. Phases 0–3 are unchanged in logic, structure, and outputs (only one descriptive forward-reference inside the Phase 3 entry was updated to point at "Phase 4 Step 2.7 / Step 6.4" instead of "Phase 5 equivalence testing"; the Phase 3 supervisor file `baseline-testing-supervisor.md` is NOT modified). The technical agent ID `refactoring-supervisor` is preserved for backward compatibility — only the human-facing capability name changes (`display_name: Application Replatforming` added to frontmatter); existing dependencies in 13 files (catalog.json, README, wiki, refactoring-tobe-supervisor, etc.) continue to resolve. Coordinated changes:
  - **Phase 4 — Application Replatforming** (the rewriting phase, redesigned). Replaces both the previous big-bang Phase 4 (TO-BE Refactoring with `scaffold-todo` and 6 internal waves) AND the previous separate Phase 5 (TO-BE Testing iterative loop). New structure: 7 sequential steps with hard build+start+test gates between every step. Step 0 Bootstrap (project scaffold + build + startup must succeed), Step 1 Minimal Runnable Skeleton (empty controllers / blank pages / security simplified), Step 2 Incremental Feature Loop (per UC: implement → tests → build → run → validate vs Phase 3 oracle — next feature only after current one passes all 7 sub-steps), Step 3 Mandatory Validation Loop (sub-loop triggered automatically on any failure — root cause + minimal fix + re-validate; blocks forward progress), Step 4 Progressive System Construction (vertical slices, no premature abstraction, app always buildable/runnable/testable), Step 5 Hardening (Spring Security 6 + env config + RFC 7807 + JSON logging + CSP — re-validated after each change), Step 6 Final Validation (full test suite + business-flow validation vs Phase 3 oracle + TODO sweep + deliverable `01-replatforming-report.md` with PO sign-off, blocked while critical/high failures or pending TODOs remain).
  - **Driver model change**: Phase 4 is no longer dispatched as a single Agent call to a black-box sub-supervisor. The Workflow Supervisor drives the 7-step loop directly using fine-grained sub-agents (`developer-java-spring`, `developer-frontend`, `test-writer`, `debugger`, `code-reviewer`, `api-designer`, `software-architect`) because the per-step / per-feature gates require workflow-level observation. The legacy `refactoring-tobe-supervisor` and `tobe-testing-supervisor` agent files remain on disk for backward compatibility but are NOT in the canonical Phase 4 path.
  - **Workflow manifest schema**: the old separate `phase 4` (with `code_scope`, `todo_count_*`, `todo_budget_status`) and `phase 5` (with `iterations[]`, `iteration_count`, `current_pass_rate_percent`, `trend`, `po_signoff`) entries are replaced by a single new `phase 4` entry with `current_step` (0–6), `steps[]` (one entry per step with `gate` evidence + `feature_loop_progress` for Step 2 + `hardening_concerns_done` for Step 5 + `po_signoff` for Step 6), and `validation_loop_total_triggers` (cumulative Step 3 sub-loop count).
  - **Per-step recap (replaces old E.4 TODO budget + E.5 iteration recap)**: new Step E.4 specifies four recap shapes — gate-step recap (Steps 0/1/5/6), per-feature recap (Step 2.7 each feature), Step 3 sub-loop convergence recap, and end-of-Phase-4 recap (Step 6 + PO sign-off). Each recap surfaces hard-gate evidence (build green / app starts / tests pass / behavior validated) and asks the user `[next | pause | revise | stop]` (Step 2) or `[yes | revise | stop]` (gate steps).
  - **Decision rules**: 11 new rows added (Phase 4 Step 0/1/2/3/4/5/6 failure handling, sub-loop divergence after 3 attempts, PO sign-off block while critical/high remain, TODO sweep block, refusal to skip-ahead a Phase 4 step). The 7 prior rules tied to Phase 4 TODO budget + Phase 5 iteration loop are removed.
  - **Constraints**: 4 new invariants codified — "the application is always in a working state", "no skipping a failing gate", "Phase 4 ends with PO sign-off OR `partial` state", "there is no separate Phase 5". Constraint allowing the orchestrator to invoke fine-grained sub-agents (vs only phase supervisors) is now explicit and limited to Phase 4.
  - **Activation examples**: trigger phrases extended to "application replatforming" + Italian; the legacy "Run Phase 5 / Validate TO-BE / Run equivalence verification" entries are replaced with a clarification that there is no separate Phase 5 anymore.
- **`refactoring-supervisor` bumped to 2.2.0 — Phase 4 low-TODO directive + Phase 5 iterative loop.** Driven by user feedback that Phase 4 was ending with too many TODOs and Phase 5 ran as a one-shot instead of cycling on test failures. Two coordinated changes:
  - **Phase 4** — default code-scope flipped from `scaffold-todo` to `full`. The Workflow Supervisor now appends a TODO-budget directive to its dispatch prompt: soft target `≤ 1 TODO per 10 use cases`, every TODO must carry `ambiguity_reason` + AS-IS source reference, stylistic skeleton TODOs forbidden. The post-phase recap gains a new **Step E.4 TODO budget audit block** that surfaces total / per-UC TODO counts and the top-10 unfilled branches with their ambiguity reason; if the budget is exceeded the phase is classified as `partial` and `proceed` is refused unless the user explicitly overrides. Workflow manifest schema gains `code_scope`, `todo_count_total`, `todo_count_per_uc`, `todo_budget_status` for Phase 4.

The Phase 5 schematic gains an explicit ITERATION LOOP block at the top.
